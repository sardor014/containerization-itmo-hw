## Разбор Bad Practices

В файле `bad_example/Dockerfile` были допущены 3 ошибки

### 1. Тег `latest` в базовом образе
* **Плохая практика:** `FROM jupyterhub/jupyterhub:latest`
Тег `latest` не является детерминированным. При внезапном обновлении версии билд может упасть из-за несовместимости вресий или отсутствия используемых модулей.
* **Как исправлено:** Зафиксирована конкретная версия `FROM jupyterhub/jupyterhub:3.1.1`. Это гарантирует воспроизводимость билда в любое время.

### 2. Множественные слои `RUN` и лишние пакеты
* **Плохая практика:** Использование отдельных команд `RUN` для установки каждого пакета `vim`, `git`, `nano` (плюс эти пакеты еще и не нужны).
Каждая инструкция `RUN` создает новый неизменяемый слой в файловой системе образа, что увеличивает итоговый размер образа.
* **Как исправлено:** Лишние пакеты (взятые из pdf файла с описанием лабы) удалены. Если бы установка требовалась, команды были бы объединены в один `RUN` через `&&`.

### 3. Относительный путь в `WORKDIR`
* **Плохая практика:** `WORKDIR jupyter_folder`
Docker интерпретирует этот путь относительно текущей рабочей директории. Если в базовом образе рабочая директория была изменена (например, на `/srv`), то папка создастся по пути `/srv/jupyter_folder`
* **Как исправлено:** Использован абсолютный путь `WORKDIR /jupyter`. Папка всегда будет создана в корне файловой системы контейнера.

## Когда не надо использовать контейнеры

1.  **Старые монолитные приложения**

2.  **Приложения с высокими требованиями к изоляции безопасности** - из контейнера можно сбежать и получить доступ к хост-системе
